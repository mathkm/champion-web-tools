# 変更履歴 2026-07-16

## realtime版：初期読込を直近1週間に限定し、過去日への遡り取得を追加

### 対象ファイル
- `html/ampere/realtime.html`

### 背景・目的
起動時に `/ampere/data/` 配下の全CSV（実データで85ファイル・約205MB）を無条件に読み込んでおり、
データ蓄積とともに初期表示のダウンロード量とメモリ使用量が青天井に増え続けていた。
初期表示を「現在日時から直近1週間」に限定し、それより過去のデータを見たいときだけユーザー操作を
きっかけに遡って追加取得する方式に変更した。

「全期間」プリセットは読込済みの範囲のみを対象とする（サーバーへの追加取得はしない）現行動作のまま
とする。遡り取得に件数上限は設けない。初期ウィンドウの日数（1週間）はコード先頭の定数で固定し、UIは
増やさない。以上はいずれも実装前にユーザーと確認済みの仕様判断。

ファイル名 `YYYYMMDD_HHMMSS_mmm.csv` は先頭行の受信日時と完全に一致することを実データで確認済みで、
ファイル名だけで各ファイルの開始時刻・守備範囲を判定できる（中身を読む必要がない）。

### 変更内容

すべて `<script>` 内の IIFE（旧265行目〜）への変更。

#### 1. 定数と状態の追加
`MAX_POINTS` の近くに定数を、`fileState` などの近くに状態変数を追加した。

```js
const INITIAL_WINDOW_DAYS = 7;                 // 初期読込ウィンドウ(現在日時からの遡り日数)
const FILENAME_TS_RE = /^(\d{4})(\d{2})(\d{2})_(\d{2})(\d{2})(\d{2})_(\d{3})\.csv$/i;
...
let loadWindowStartMs = Date.now() - INITIAL_WINDOW_DAYS * 24 * 3600 * 1000;
let backfillPending = false;
```

#### 2. ファイル名 → 開始時刻のパース
`parseDateTime()` / `fromLocalInputValue()` と同じ流儀（`new Date(文字列)` に頼らず正規表現で分解して
ローカル時刻の `Date` を組み立てる）で `parseFileStartTime()` を追加した。

#### 3. ウィンドウ内ファイルの選別
各ファイルの守備範囲を `[自身の開始時刻, 次ファイルの開始時刻)` とみなし、`windowStartMs` 以降のデータを
含みうるファイルだけを返す `selectFilesInWindow()` を追加した。境界を跨ぐファイル（ウィンドウ開始時刻を
内包するファイル）も取りこぼさないよう選ばれる。行単位の時刻フィルタは行わない（`fileState` のバイト
オフセットが進んでしまい、後の遡り取得で読み直せなくなるため）。

#### 4. `poll()` の取り込み対象を選別結果に差し替え
```js
const files = await fetchListing();
files.sort((a, b) => NAME_COLLATOR.compare(a.name, b.name));
const targets = selectFilesInWindow(files, loadWindowStartMs);   // ← 追加
let added = 0;
for (const f of targets) { /* ...既存のまま... */ }
```
ウィンドウ外のファイルには `fileState` のエントリを一切作らない（作ると後から遡り取得しても「既読」と
判定されて読み込まれなくなるため）。

#### 5. 遡り取得のトリガー
`requestBackfill(start)` を追加し、`onRangeInputChange()` の末尾（`setRange(start, end, 'input')` の直後）
で呼ぶようにした。プリセットボタンのハンドラは変更していない（「全期間」は読込済み範囲のみという仕様
のため）。

```js
function requestBackfill(start) {
  if (!start || start.getTime() >= loadWindowStartMs) return; // 既にカバー済み
  loadWindowStartMs = start.getTime();
  if (polling) { backfillPending = true; return; }
  setStatus('loading');
  poll();
}
```

#### 6. `poll()` の finally で持ち越し要求を消化
```js
} finally {
  polling = false;
  if (backfillPending) { backfillPending = false; poll(); }
  else scheduleNext();
}
```

#### 7. `setStatus()` に読込中の表示を追加
```js
function setStatus(kind, err) {
  statusText.classList.remove('ok', 'error');
  if (kind === 'loading') { statusText.textContent = '過去データ読込中…'; return; }
  // ...既存のまま...
}
```

### 触っていない箇所（意図的に維持）
- `downsample()` / `MAX_POINTS`
- `setRange()` / `refreshView()` / `renderChart()` の relayout 競合対策ロジック（「ライブ追従時に開始が
  リセットされる」バグの修正箇所）。初期表示の「開始」は `refreshView()` が空の入力欄に対し
  `setRange(null, null, 'live')` → `dataStartT()` を使うことで自動的に約1週間前になるため、初期値を
  明示代入するコードは追加していない。
- `classifyRow()` / `ingestChunk()` のバイトオフセット差分ロジック
- 統計計算・積算のロジック

### 動作確認

#### 1. 選別ロジックの単体確認（Node、ブラウザ不要）
`data/` にある実ファイル85件（`20260707_103722_500.csv` 〜 `20260715_093845_059.csv`）の名前一覧に対し、
本番コードと同一の `parseFileStartTime` / `selectFilesInWindow` をスクラッチパッド上のNodeスクリプトで
実行して確認した。

- 全85件が `parseFileStartTime` でパースでき、`null` は0件（OK）。
- `windowStartMs = 2026-07-09 00:00:00` を与えたとき、`20260707_*`（7件）は全て除外。`20260708_*`
  （9件）は8件除外・**1件のみ含まれる**（`20260708_224825_401.csv`）。これはバグではなく、この
  ファイルの守備範囲が `[22:48:25.401, 2026-07-09 00:00:00.486)` であり、`windowStartMs`
  ちょうど（00:00:00.000）を内包するため、「境界を跨ぐファイルも選ばれる」という設計どおりの挙動。
  実データの次ファイル `20260709_000000_486.csv` の開始が厳密に00:00:00でなく486ms後ろにずれている
  ことに起因する、想定内の誤差（プランの検証節の例示値は簡略化されており、この1ファイル分は考慮され
  ていなかった）。
- `windowStartMs = 2026-07-09 12:00:00`（ファイル境界を跨ぐ時刻）のとき、その時刻を内包する
  `20260709_095058_048.csv`（`20260709_121333_282.csv` の1つ手前）が含まれることを確認（OK＝境界跨ぎ
  ファイルを落とさない要件の核）。
- `windowStartMs = 0` で全85件が選ばれることを確認（OK）。

#### 2. ブラウザでの実地確認（Docker + `champion-web-tools/data` の実ファイルをマウント）
スクラッチパッドに `docker-compose.override.yml` を作り（`docker-compose.yml` 本体は変更せず）、
`web` サービスのデータボリュームを `C:\dev\champion-web-tools\data` に差し替えて
`http://localhost:8081/ampere/realtime/` を実機（Chromiumベースのブラウザペイン）で確認した。
検証時の実行日時は2026-07-16 10:20頃、ローカルデータは2026-07-07〜2026-07-15の85ファイル。

- **初期表示**：ファイル一覧に載ったのは62/85件で、最古が `20260709_095058_048.csv`
  （「開始」欄も `2026-07-09 09:50:58`）。DevTools Networkで `20260707_*` / `20260708_*` への
  リクエストが1件も発生していないことを確認（OK＝ウィンドウ外を読みに行かない）。
- **遡り取得**：「開始」欄を `2026-07-07T00:00:00` に変更 → Networkに `20260707_*`（7件）・
  `20260708_*`（8件）・未読分の`20260709_*`のCSVリクエストが発生し、ファイル数が62→85、読込点数が
  896,316→1,100,736 に増加。グラフのx軸左端が Jul 8 目盛りまで伸びたことを確認。
- **「開始」欄の保持**：遡り取得の前後、および約10秒後の自動ポーリング（ライブ追従ON）を1サイクル
  挟んだ後も「開始」欄の値がリセットされず保持されていることを確認（OK＝既知の「ライブ追従で開始が
  戻る」バグの再発なし）。
- **「全期間」ボタン**：クリック後もNetworkに新規CSVリクエストが発生しないことを確認（読込済み範囲
  へのズームのみ、というプランの仕様どおり）。表示期間が読込済みの全範囲
  （2026/07/07 10:37:22 〜 2026/07/15 11:28:08）にズームされた。
- ブラウザConsoleにエラー・警告は一貫して出ていないことを確認。
- 確認後、検証用コンテナは `docker compose ... down` で停止・削除済み。スクラッチパッドの
  Nodeスクリプト・compose overrideはリポジトリに持ち込んでいない。

### 備考・既知の制約
- `loadWindowStartMs` は起動時に一度だけ決まる下限であり、スライディングウィンドウではない。ページを
  開いたまま数日経つと読込済み範囲は1週間より広がるが、既読データを捨てる実装は入れていない（意図的）。
- 「全期間」ボタンは読込済み範囲へのズームのみで、サーバーへの追加取得は行わない。
- 境界を跨ぐファイルが選ばれる仕様上、ウィンドウ開始より最大でファイルローテーション間隔（約2〜3時間）
  ほど古いデータが混ざることがある（「1週間前からのデータが確実に揃う」ことを優先した設計）。
